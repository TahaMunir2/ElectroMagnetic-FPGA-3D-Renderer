SECTION 3 — WAVE INJECTION

3. Wave Injection — How It Works
3.1 What Wave Injection Means Physically
In a real EM experiment, a wave is generated by an oscillating current in an antenna or an
oscillating voltage on an electrode. In your FDTD simulation, there is no real antenna — you
create a wave by forcing one or more grid cells to follow a prescribed time-varying waveform. This
is called source injection.
The physical analogy is exact: your conducting paper electrodes driven by the AD9833 are the
physical source. In the simulation, grid cells at positions corresponding to those electrode
positions are forced to oscillate — the simulation source mirrors the physical source. This is how
the FDTD simulation and the physical measurement stay synchronised.

3.2 The Hard Source — Simplest Approach
3.2.1 What It Does
A hard source completely overrides the FDTD update equation at the source cell. Every time step,
after the FDTD update runs across the whole grid, the source cell is simply set to a prescribed
value regardless of what the FDTD equation computed:
Ex[isrc][jsrc] = A × sin(2π × f₀ × n × Δt)

Where n is the time step counter, f₀ is the normalised source frequency, A is amplitude, and Δt is
the time step.

3.2.2 Why It Causes Problems
A hard source reflects incoming waves. If a wave propagates outward from the source, reflects
off a boundary or obstacle, and returns to the source cell, it hits the fixed value and reflects again
— creating a non-physical double-reflection artefact. For a simple demo this is acceptable. For
quantitatively accurate results, use a soft source.

3.3 The Soft Source — Physically Correct
3.3.1 What It Does
A soft source adds the source waveform to the FDTD update result rather than replacing it:
Ex[isrc][jsrc] += A × sin(2π × f₀ × n × Δt)

This allows reflected waves to pass back through the source cell without artificial scattering. The
source drives the field but does not block wave propagation through its location. This is the
standard approach for antenna simulation.

3.3.2 Verilog — Modification to FDTD Engine

In the FDTD sequencer, after computing the Ex update for the source cell, add the source value
before writing to BRAM:

The CORDIC input is a phase angle in the range -π to +π, represented as a 18-bit fixed-point

// Phase accumulator (Q5.13 to handle range > 2*pi before wrapping)
reg signed [24:0] phase_accum;
// f0 * 2pi * dt in Q3.13:
// f0=0.05, dt=0.35, 2pi=6.283
// increment = 0.05 * 6.283 * 0.35 = 0.10995 -> 0.10995 * 8192 = 901

= 25'sd25736;
localparam signed [24:0] TWO_PI
= 25'sd51472;

// 2*pi * 8192

phase_accum <= 0;
else begin

// Wrap at +pi

phase_accum <= phase_accum - TWO_PI;
end

// Feed to CORDIC (take lower 18 bits as Q3.13 phase)
assign cordic_phase_in = phase_accum[17:0];

3.7 How the Physical Electrode Connects to the Simulation
Source
The AD9833 drives the physical electrode on the conducting paper at 1kHz. Simultaneously, the
FPGA simulation runs its own source at normalised frequency f₀. These are two independent

things — the physical electrode drives the real paper, the simulation source drives the virtual grid.
They are connected conceptually (same position, same waveform shape) but not electrically.
The connection happens through boundary conditions: the probe measures the physical potential
at the electrode position, and this value is written to the corresponding simulation grid cell. The
physical measurement anchors the simulation to reality — the simulation field at the electrode
must match what you physically measure.

SECTION 4 — MEASURING THE WAVE

4. Measuring the Wave
4.1 What You Are Actually Measuring
There are two completely separate measurement systems in this project:
•

The conducting paper probe: measures the static or quasi-static potential V(x,y) at any
point on the paper surface — this is the Laplace solution, used as boundary condition for
the FDTD simulation.
• The FDTD field arrays: the simulation computes Ex, Ey, Bz at every grid point — these
are the wave fields, not directly measurable from the paper (the paper operates at too
low a frequency to show wave propagation).
The FDTD simulation IS the wave measurement. The physical paper gives you the boundary
conditions. The FPGA computes how an EM wave would behave given those boundary
conditions. You display the computed wave fields in 3D.

4.2 Measuring the Physical Potential — The Probe Circuit
4.2.1 What the Probe Sees
The conducting paper with two electrodes has a potential distribution across its surface. The
probe tip (a fine metal point) touches the paper at any (x,y) position and sits at the potential V(x,y)
of that point. This potential needs to be measured by the Pynq's XADC without disturbing the field
— same loading problem as the water tray, same solution: a JFET input op-amp buffer.

4.2.3 Using AC Drive with Lock-In on the Paper
If you use AC drive (1kHz from AD9833) on the conducting paper instead of DC, you get the same
noise rejection benefits as with water. The probe then sees a 1kHz AC signal whose amplitude
encodes the potential. The lock-in demodulation extracts this amplitude cleanly.
For the conducting paper with a handheld probe, lock-in is optional — paper has no electrolysis
problem and DC is stable. But if lab noise is an issue, switch to AC drive and add the lock-in
demodulation for the probe channel:
Python lock-in for probe measurement
import numpy as np
from pynq import MMIO
f0 = 1000
fs = 20000
N = 20

# drive frequency Hz
# ADC sample rate Hz
# samples per cycle

n = np.arange(N)
ref_cos = np.cos(2 * np.pi * f0 * n / fs)
ref_sin = np.sin(2 * np.pi * f0 * n / fs)
def measure_probe_voltage():
# Collect N samples from XADC channel 0
samples = np.array([read_xadc(channel=0) for _ in range(N)])
# Lock-in demodulation
I = np.mean(samples * ref_cos)
Q = np.mean(samples * ref_sin)
amplitude = 2.0 * np.sqrt(I**2 + Q**2)
return amplitude # volts, noise-free

4.3 XY Position Sensing — Resistive Touch Panel
4.3.1 How a Resistive Touch Panel Works

A resistive touch panel has four layers: two transparent resistive sheets (ITO — Indium Tin Oxide)
separated by a small air gap, with non-conductive spacer dots. When pressed, the two sheets
make contact at the touch point.
To measure X position: apply voltage across the top sheet left-to-right, measure voltage at the
contact point on the bottom sheet. The voltage is proportional to the X position — it is a simple
resistive voltage divider. To measure Y position: apply voltage across the bottom sheet top-tobottom, measure on the top sheet.
The panel has four connections: X+, X-, Y+, Y-. You drive two and measure two alternately.

4.3.2 Connecting to the Pynq
Touch panel connections to Pynq Arduino header

X+ → D2 (GPIO output — drive high or float)
X- → D3 (GPIO output — drive low or float)
Y+ → A1 (XADC channel 1 — measure X position voltage)
Y- → D4 (GPIO output — drive high or float)
For Y measurement:
Y+ → drive high (D4 high)
Y- → drive low (implicit via resistor to GND)

4.4 Reading the XADC from Python
The Pynq's XADC Wizard IP exposes its outputs as memory-mapped registers. The Python MMIO
class reads them directly:
xadc_read.py
from pynq import Overlay, MMIO
# Load your bitstream first
ol = Overlay('/home/xilinx/em_fdtd.bit')
# Access XADC via its AXI-Lite register interface
# Base address from Vivado address editor (typically 0x43C00000)
xadc = MMIO(0x43C00000, 0x10000)
def read_xadc(channel):
# XADC status register addresses:
# Channel 0 (VP/VN) = offset 0x200
# Channel 1 (VAUX1) = offset 0x210
# Channel 2 (VAUX9) = offset 0x218
offsets = {0: 0x200, 1: 0x210, 2: 0x218}
raw = xadc.read(offsets[channel])

# Raw value is 16-bit left-justified: bits [15:4] are the 12-bit result
adc_value = (raw >> 4) & 0xFFF
# 0 to 4095
voltage = adc_value * 3.3 / 4096 # convert to volts
return voltage

Reading 65,536 values one by one over AXI-Lite is slow (~0.5 seconds for one field). For the
demo display, the FPGA reads the BRAM directly and renders — the CPU only reads the field for
debugging or for the comparison plot. Use DMA for fast bulk transfers if you need the full field in
Python at runtime.

7.4 Verifying FDTD is Correct
Before the demo you must verify the simulation is physically correct. Three checks:
32. Energy conservation: with no source and no PML, run for 1000 steps. The total field
energy ∑(Ex²+Ey²+Bz²) should not change. Any change indicates a bug in the update
equations.
33. Wave speed: inject a single pulse. Measure how many time steps it takes to travel from
the source to a point 50 cells away. Speed should be c_eff × Δx/Δt = 0.5 cells/step. If 50
cells takes 100 steps: speed = 0.5 ✓
34. Wavelength: with continuous source at f₀ = 0.05, wavelength should be c_eff/f₀ =
0.5/0.05 = 10 cells. Count the distance between wavefronts visible on screen — should
be 10 pixels.

ad9833_spi.py — program sine wave frequency
from pynq.lib import MicroblazeLibrary
from pynq import Overlay
# Access SPI on Pmod connector
# Pynq provides SPI via the Arduino SPI pins or Pmod
import spidev
spi = spidev.SpiDev()
spi.open(1, 0) # bus 1, device 0
spi.max_speed_hz = 1000000
spi.mode = 0b10 # SPI mode 2 for AD9833
def ad9833_set_frequency(freq_hz, mclk=25e6):
"""Program AD9833 to output sine wave at freq_hz"""
freq_word = int(freq_hz * (2**28) / mclk)
msb = (freq_word >> 14) & 0x3FFF
lsb = freq_word
& 0x3FFF
# Write control: reset, then freq LSB, MSB, then run
def write16(val):
spi.xfer2([(val >> 8) & 0xFF, val & 0xFF])
write16(0x2100)
# reset
write16(0x4000 | lsb)
# FREQ0 LSB
write16(0x4000 | msb)
# FREQ0 MSB
write16(0x2000)
# release reset, sine output
ad9833_set_frequency(1000)
print('AD9833 set to 1kHz')

# 1kHz

Check CB_B: a wave should travel exactly c_eff = 0.5 cells per time step. After 100 time steps
